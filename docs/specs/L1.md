# SeamQ — L1 Requirements (High-Level)

## L1-1: Workspace Scanning

SeamQ shall accept one or more Angular workspace root paths and statically parse their project structures, TypeScript source files, Angular metadata, and configuration files to build a comprehensive model of each workspace's exports, imports, services, components, and type definitions.

## L1-2: Interface Seam Detection

SeamQ shall analyze parsed workspaces to detect interface boundaries (seams) where types, services, tokens, bindings, events, routes, state, or messages cross workspace boundaries — classifying each seam by type (Plugin Contract, Shared Library, Message Bus, Route Contract, State Contract, HTTP/API Contract) with a confidence score.

## L1-3: ICD Document Generation

SeamQ shall generate complete Interface Control Documents (ICDs) equivalent in depth, structure, and completeness to a hand-authored aerospace-grade ICD — containing introduction, reference documents, interface definition (overview, scope of responsibility, standards, layer-by-layer contracts, data objects, lifecycle/state, consumption patterns, timing), protocols, appendices (glossary, TBD items, traceability matrix, checklist trace, diagram index), and revision history.

## L1-4: Diagram Generation

SeamQ shall generate PlantUML diagrams covering class diagrams (12 types), sequence diagrams (15 types), state diagrams (2 types), and C4 architecture diagrams (12 types) — outputting `.puml` source files and optionally rendering to SVG/PNG via a PlantUML server.

## L1-5: CLI Interface

SeamQ shall provide a .NET CLI (distributed as a `dotnet tool` via NuGet) with commands: `scan`, `list`, `generate`, `diagram`, `inspect`, `validate`, `diff`, `init`, `export`, and `serve` — supporting global flags (`--verbose`, `--quiet`, `--no-color`, `--output-dir`, `--config`) and using the `System.CommandLine` framework with a file-per-command pattern and Microsoft.Extensions for DI, logging, and configuration.

## L1-6: Configuration System

SeamQ shall support configuration via a `seamq.config.json` file and/or CLI flags — covering workspace definitions (path, alias, role), seam filtering (include/exclude types, custom decorators), output settings (directory, formats, diagram rendering options, themes), analysis options (maxDepth, followNodeModules, confidenceThreshold), and ICD metadata (title, document number, revision, classification, template).

## L1-7: Baseline Diffing

SeamQ shall support saving scan results as baselines and comparing subsequent scans against a baseline to produce a change report identifying additions, modifications, and removals in the contract surface.

## L1-8: Contract Validation

SeamQ shall validate that consumer workspaces correctly implement provider contracts — checking interface implementations, injection token provisioning, and input/output binding compliance — and report errors and warnings.

## L1-9: Data Export

SeamQ shall export raw seam data as structured JSON for integration with external toolchains.

## L1-10: Performance and Reliability

SeamQ shall meet performance targets (500 files < 30s, 2000 files < 2min, ICD gen < 5s, diagrams < 10s, memory < 1GB), support graceful degradation when workspaces fail to parse, produce deterministic output, and return appropriate exit codes (0=success, 1=partial failure, 2=fatal error).

## L1-11: Cross-Platform Compatibility

SeamQ shall run on Windows, macOS, and Linux on .NET 8+ and support Angular 15–19, Nx 16–19, and TypeScript 5.0+ workspaces.

## L1-12: Security and Privacy

SeamQ shall perform read-only file system access on scanned workspaces, make no network calls unless PlantUML server rendering is configured, and ensure scanned source code never leaves the local machine.
