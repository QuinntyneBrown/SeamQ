# SeamQ — L2 Requirements (Detailed)

---

## L1-1: Workspace Scanning

### L2-1.1: CLI Workspace Path Acceptance

**Traces to:** L1-1

SeamQ shall accept one or more file paths to Angular workspace roots via CLI arguments to the `scan` command.

**Acceptance Criteria:**
- `seamq scan <path1> [path2] [pathN]` accepts 1+ paths
- Paths that do not exist produce a clear error message with the invalid path
- Relative and absolute paths are both supported
- Paths from config file are used when no CLI paths are provided

### L2-1.2: Workspace Type Auto-Detection

**Traces to:** L1-1

SeamQ shall auto-detect the workspace type: Nx monorepo, Angular CLI workspace, or standalone project.

**Acceptance Criteria:**
- Detects Nx monorepo by presence of `nx.json`
- Detects Angular CLI workspace by presence of `angular.json` without `nx.json`
- Detects standalone project by presence of `angular.json` with a single project
- Returns detected type in scan output

### L2-1.3: Project Configuration Parsing

**Traces to:** L1-1

SeamQ shall parse `angular.json`, `project.json`, `nx.json`, and `workspace.json` to discover projects and libraries within each workspace.

**Acceptance Criteria:**
- Parses Angular CLI `angular.json` projects array
- Parses Nx `project.json` per-project configuration
- Parses `nx.json` workspace-level settings and plugin configuration
- Parses `workspace.json` (legacy Nx format)
- Discovers all projects and classifies them as application or library

### L2-1.4: TypeScript Path Resolution

**Traces to:** L1-1

SeamQ shall resolve TypeScript project references and path aliases from `tsconfig.json` and `tsconfig.base.json`.

**Acceptance Criteria:**
- Parses `compilerOptions.paths` from tsconfig files
- Resolves path aliases to actual file system paths
- Follows `references` array for project references
- Supports `tsconfig.base.json` inheritance via `extends`

### L2-1.5: Public API Barrel Parsing

**Traces to:** L1-1

SeamQ shall parse public API barrel files (`public-api.ts`, `index.ts`) to determine the exported surface of each library.

**Acceptance Criteria:**
- Identifies barrel files by convention (`public-api.ts`, `index.ts`)
- Parses `export * from`, `export { X } from`, and `export default` statements
- Builds a complete list of publicly exported symbols per library
- Handles re-exports through multiple barrel levels

### L2-1.6: Path Exclusion Support

**Traces to:** L1-1

SeamQ shall support glob patterns and a `.seamqignore` file for excluding paths from scanning.

**Acceptance Criteria:**
- `.seamqignore` file follows `.gitignore` syntax
- `node_modules` excluded by default
- CLI `--exclude` flag accepts glob patterns
- Config `analysis.exclude` array is supported

### L2-1.7: AST Caching

**Traces to:** L1-1

SeamQ shall cache parsed AST results for subsequent runs with file-hash-based invalidation.

**Acceptance Criteria:**
- Cache stored in `.seamq/cache` directory
- Cache key is SHA-256 hash of file content
- Subsequent scans skip unchanged files
- `--no-cache` flag disables caching

### L2-1.8: Bootstrap File Parsing

**Traces to:** L1-1

SeamQ shall parse `main.ts` / bootstrap files to discover provider registrations, multi-provider tokens, and plugin factory functions.

**Acceptance Criteria:**
- Detects `bootstrapApplication()` and `bootstrapModule()` calls
- Extracts provider arrays from bootstrap configuration
- Identifies `multi: true` provider registrations
- Identifies `provideXxx()` factory function calls

### L2-1.9: Injectable Service Parsing

**Traces to:** L1-1

SeamQ shall parse `@Injectable()` service files to extract full method/property API signatures, observable/signal types, and TSDoc documentation.

**Acceptance Criteria:**
- Detects `@Injectable()` decorated classes
- Extracts all public method signatures (name, parameters, return type)
- Extracts all public property signatures (name, type, readonly/signal/computed)
- Extracts Observable/Subject/BehaviorSubject typed properties with generic type parameters
- Extracts TSDoc/JSDoc comments for methods and properties

### L2-1.10: Component Parsing

**Traces to:** L1-1

SeamQ shall parse component files to extract `@Input()`, `@Output()`, signal inputs, lifecycle hook implementations, and `inject()` calls.

**Acceptance Criteria:**
- Detects `@Component()` decorated classes
- Extracts `@Input()` bindings with type, default value, and alias
- Extracts `@Output()` bindings with event type
- Extracts `input()`, `input.required()`, and `model()` signal inputs
- Detects lifecycle hook implementations (OnInit, OnDestroy, OnChanges, etc.)
- Detects `inject()` function calls and constructor injection

### L2-1.11: Model/Interface Parsing

**Traces to:** L1-1

SeamQ shall parse model/interface files to extract full interface definitions, enums, type aliases, and generic type parameters with field-level detail.

**Acceptance Criteria:**
- Extracts interface declarations with all properties (name, type, optional flag)
- Extracts enum declarations with all members and values
- Extracts type aliases including union and intersection types
- Extracts generic type parameters and constraints
- Extracts TSDoc/JSDoc field-level documentation

### L2-1.12: State Management Parsing

**Traces to:** L1-1

SeamQ shall parse store/state management files to extract signal-based state shapes, computed signals, mutation methods, and persistence patterns.

**Acceptance Criteria:**
- Detects `signal()`, `computed()`, `effect()` usage
- Extracts signal state shapes and types
- Detects localStorage/sessionStorage persistence patterns
- Extracts NgRx store patterns (actions, selectors, reducers) when present

### L2-1.13: TSDoc Extraction

**Traces to:** L1-1

SeamQ shall extract TSDoc/JSDoc comments from all analyzed files to populate ICD descriptions, parameter documentation, and constraint annotations.

**Acceptance Criteria:**
- Extracts `/** */` block comments preceding declarations
- Parses `@param`, `@returns`, `@example`, `@deprecated` tags
- Associates documentation with the correct symbol
- Preserves markdown formatting within comments

## L1-2: Interface Seam Detection

### L2-2.1: Cross-Workspace Interface Detection

**Traces to:** L1-2

SeamQ shall detect TypeScript interfaces and abstract classes exported from one workspace and implemented/extended in another.

**Acceptance Criteria:**
- Identifies interfaces exported from workspace A's public API
- Detects classes in workspace B that implement those interfaces
- Detects classes in workspace B that extend abstract classes from workspace A
- Creates a seam record linking provider and consumer workspaces

### L2-2.2: Injection Token Detection

**Traces to:** L1-2

SeamQ shall detect Angular `InjectionToken` instances defined in one workspace and provided/injected in another, including multi-provider patterns.

**Acceptance Criteria:**
- Detects `new InjectionToken()` declarations
- Traces token usage across workspace boundaries (provided in one, injected in another)
- Identifies `multi: true` provider registrations
- Detects `@Inject()` decorator and `inject()` function referencing cross-workspace tokens

### L2-2.3: Input/Output Binding Detection

**Traces to:** L1-2

SeamQ shall detect `@Input()` and `@Output()` bindings on components exported across workspace boundaries, with types, defaults, and TSDoc descriptions.

**Acceptance Criteria:**
- Detects `@Input()` decorated properties with type and default
- Detects `@Output()` decorated properties with EventEmitter type parameter
- Links bindings to components in the public API surface
- Includes TSDoc descriptions when available

### L2-2.4: Signal Input Detection

**Traces to:** L1-2

SeamQ shall detect Angular signal inputs (`input()`, `input.required()`) and model signals (`model()`) on exported components.

**Acceptance Criteria:**
- Detects `input()` calls with type parameter and default
- Detects `input.required()` calls with type parameter
- Detects `model()` calls for two-way binding signals
- Includes these in the contract surface of exported components

### L2-2.5: Observable/Subject Detection

**Traces to:** L1-2

SeamQ shall detect RxJS Subject/BehaviorSubject/ReplaySubject instances and Observable-typed properties exposed as public API, including generic type parameters.

**Acceptance Criteria:**
- Detects `Subject<T>`, `BehaviorSubject<T>`, `ReplaySubject<T>` typed properties
- Detects `Observable<T>` typed public properties and method return types
- Extracts the generic type parameter `T`
- Flags these as message bus seam candidates when crossing workspace boundaries

### L2-2.6: Route Contract Detection

**Traces to:** L1-2

SeamQ shall detect route contract seams: `loadChildren`, `loadComponent`, route data interfaces, and route guard contracts.

**Acceptance Criteria:**
- Detects `loadChildren` and `loadComponent` in route configurations
- Identifies route data interface contracts
- Detects route guard implementations that cross workspace boundaries
- Detects `CanActivate`, `CanDeactivate`, `Resolve` guards

### L2-2.7: NgRx Integration Detection

**Traces to:** L1-2

SeamQ shall detect NgRx integration seams: exported actions, selectors, feature state interfaces, and effect triggers.

**Acceptance Criteria:**
- Detects `createAction()` exports crossing workspace boundaries
- Detects `createSelector()` exports
- Detects feature state interfaces shared across workspaces
- Detects effect classes that respond to cross-workspace actions

### L2-2.8: SignalR Hub Detection

**Traces to:** L1-2

SeamQ shall detect SignalR hub method signatures, hub connection patterns, and message types shared across workspaces.

**Acceptance Criteria:**
- Detects SignalR hub proxy method calls (`.invoke()`, `.on()`)
- Extracts method names and parameter types
- Identifies shared message types used in hub communication
- Links hub contracts to the workspaces that use them

### L2-2.9: ForRoot/ForChild Configuration Detection

**Traces to:** L1-2

SeamQ shall detect Angular `forRoot()` / `forChild()` configuration interfaces.

**Acceptance Criteria:**
- Detects static `forRoot()` methods on NgModule classes
- Extracts configuration parameter interfaces
- Detects `forChild()` methods and their configuration
- Links these to the consuming workspaces

### L2-2.10: Service API Extraction

**Traces to:** L1-2

For each `@Injectable()` service crossing a seam boundary, SeamQ shall extract the complete public API: method signatures, property signatures, and Observable/Signal emissions.

**Acceptance Criteria:**
- Extracts method name, parameter names/types, return type for all public methods
- Extracts property name, type, and modifier (readonly/signal/computed) for all public properties
- Detects inter-service delegation relationships
- Groups services by purpose (real-time data, metadata, state, config, communication)

### L2-2.11: Factory/Provider Function Detection

**Traces to:** L1-2

SeamQ shall detect factory/provider functions (`provideXxx()` pattern) and extract their returned `Provider` configuration.

**Acceptance Criteria:**
- Detects functions matching `provide*()` naming pattern
- Extracts returned Provider token, value/class, and multi-provider flag
- Links provider functions to the workspaces that call them

### L2-2.12: Data Object Extraction

**Traces to:** L1-2

SeamQ shall extract full interface definitions, enum definitions, type aliases, and message type pairs for all types crossing seam boundaries.

**Acceptance Criteria:**
- Extracts interface field names, types, optionality, defaults, and TSDoc
- Extracts enum members with values
- Extracts type aliases, union types, intersection types, and generics
- Detects request/response message type pairs by convention

### L2-2.13: State Machine Detection

**Traces to:** L1-2

SeamQ shall detect state machines from string literal unions, enum-typed state fields, and explicit state transition methods.

**Acceptance Criteria:**
- Detects string literal union types used as state fields (e.g., `'loading' | 'loaded' | 'error'`)
- Detects enum-typed state fields
- Detects methods that transition between states
- Generates state→transition tables

### L2-2.14: Lifecycle Hook Detection

**Traces to:** L1-2

SeamQ shall detect Angular lifecycle hook implementations in boundary-crossing components and document their contract implications.

**Acceptance Criteria:**
- Detects `OnInit`, `OnDestroy`, `OnChanges`, `AfterViewInit` implementations
- Documents which hooks are required vs. optional in the contract
- Links lifecycle hooks to the component's seam contract

### L2-2.15: Protocol Classification

**Traces to:** L1-2

SeamQ shall classify each detected interface into a protocol category with sync/async classification and lifetime.

**Acceptance Criteria:**
- Classifies as: DI multi-provider, component rendering, observable push stream, signal read, request-reply, or fire-and-forget
- Determines synchronous vs. asynchronous
- Determines lifetime: one-time, long-lived, session-scoped, or short-lived
- Documents all types involved in each protocol

### L2-2.16: Seam Confidence Scoring

**Traces to:** L1-2

SeamQ shall generate a confidence score (0.0–1.0) for each detected seam based on heuristic strength.

**Acceptance Criteria:**
- Score based on strength of evidence (explicit interface impl = high, naming convention match = lower)
- Configurable threshold for minimum confidence to include
- Score displayed in `seamq list` output
- Low-confidence seams flagged with a warning

## L1-3: ICD Document Generation

### L2-3.1: Markdown ICD Output

**Traces to:** L1-3

SeamQ shall generate ICD in Markdown format with full contract surface documentation, embedded diagram image references, and TypeScript code blocks.

**Acceptance Criteria:**
- Generates a complete `.md` file per seam
- Includes all sections defined in PRD Section 6.3
- Embeds diagram references as `![alt](diagrams/path.png)` markdown images
- TypeScript interface definitions rendered as fenced code blocks
- Renders correctly on GitHub

### L2-3.2: HTML ICD Output

**Traces to:** L1-3

SeamQ shall generate ICD in HTML format with navigation, syntax highlighting, embedded diagrams, and internal cross-links.

**Acceptance Criteria:**
- Generates a complete `.html` file per seam
- Includes table of contents with anchor links
- Syntax highlighting for TypeScript code blocks
- Diagrams embedded as `<img>` tags
- Internal cross-references between sections are clickable

### L2-3.3: Introduction Section Generation

**Traces to:** L1-3

SeamQ shall generate an Introduction section with purpose, scope, system overview, applicable standards, and embedded context diagram.

**Acceptance Criteria:**
- Purpose statement derived from seam type and participants
- Scope lists the workspaces involved
- System overview summarizes the contract surface
- Applicable standards from config (`icd.standards`)
- Context diagram embedded as image reference

### L2-3.4: Reference Documents Section

**Traces to:** L1-3

SeamQ shall generate an Applicable & Reference Documents table with auto-discovered source-code references.

**Acceptance Criteria:**
- Lists file paths and line numbers for key artifacts
- Includes bootstrap files, public API barrels, model directories, service directories
- Links to configuration files (angular.json, nx.json, tsconfig.json)
- Each reference includes the workspace it belongs to

### L2-3.5: Interface Overview Section

**Traces to:** L1-3

SeamQ shall generate an Interface Definition Overview with detected API layers table and architecture diagrams.

**Acceptance Criteria:**
- Numbers each detected API layer
- Lists the purpose and key artifacts per layer
- Embeds container/architecture diagram
- Identifies provider vs. consumer for each layer

### L2-3.6: Scope of Responsibility Matrix

**Traces to:** L1-3

SeamQ shall generate a Scope of Responsibility Matrix assigning each responsibility to provider, consumer, or shared.

**Acceptance Criteria:**
- Responsibilities include: registration, discovery, rendering, input binding, subscription lifecycle, data delivery, metadata, state management, configuration, backend communication, error handling
- Each assigned to Provider, Consumer, or Shared
- Assignment derived from code analysis of who defines vs. who consumes

### L2-3.7: Registration Contract Section

**Traces to:** L1-3

SeamQ shall generate a Registration Contract section documenting injection tokens, descriptor interfaces, factory functions, bootstrap code, discovery services, and context tokens.

**Acceptance Criteria:**
- Lists all InjectionToken declarations with source references
- Documents descriptor interfaces with full type definitions
- Documents `provideXxx()` factory patterns
- Lists built-in reference implementations with source links

### L2-3.8: Component Input Contract Section

**Traces to:** L1-3

SeamQ shall generate a Component Input Contract section documenting all bindings with types, defaults, and descriptions.

**Acceptance Criteria:**
- Lists all `@Input()`, `@Output()`, signal input bindings
- Distinguishes common inputs (set by host on all plugins) from plugin-specific
- Includes full TypeScript interface definitions for configuration schemas
- Includes TSDoc descriptions for each binding

### L2-3.9: Injectable Services API Section

**Traces to:** L1-3

SeamQ shall generate an Injectable Services API section with method/property API tables grouped by purpose.

**Acceptance Criteria:**
- One subsection per service crossing the boundary
- Method table: name, parameters (name + type), return type, description
- Property table: name, type, description, observable/signal semantics
- Grouped by purpose: real-time data, metadata, state, subscription lifecycle, config CRUD, backend communication

### L2-3.10: Data Objects Section

**Traces to:** L1-3

SeamQ shall generate a Data Objects at the Boundary section organized by domain.

**Acceptance Criteria:**
- Every interface, type alias, enum, and class that crosses the seam
- Organized by domain (telemetry, configuration, state, messages, domain-specific)
- Full TypeScript interface definitions with field-level types, constraints, defaults, and documentation

### L2-3.11: Lifecycle & State Management Section

**Traces to:** L1-3

SeamQ shall generate a Lifecycle & State Management section with state machines and step-by-step sequences.

**Acceptance Criteria:**
- State machines for each stateful entity with state→transition tables
- Step-by-step sequences for key operations (add/subscribe, remove/unsubscribe)
- Each step has actor and action columns
- State diagram references embedded

### L2-3.12: Protocols Section

**Traces to:** L1-3

SeamQ shall generate a Protocols section with one subsection per distinct protocol at the seam.

**Acceptance Criteria:**
- Each protocol includes: interface type, sync/async, lifetime, types table
- State-and-sequences for each protocol
- Timing and synchronization parameters
- Full data object definitions as code blocks

### L2-3.13: Traceability Matrix

**Traces to:** L1-3

SeamQ shall generate a Requirements Traceability Matrix mapping contract elements to source files and line numbers.

**Acceptance Criteria:**
- Each contract element mapped to source file, line number, and workspace
- Supports external requirement IDs from configuration
- Matrix sortable by source file, workspace, or requirement ID

### L2-3.14: Diagram Index

**Traces to:** L1-3

SeamQ shall generate a Diagram Index listing all generated diagrams by type with file paths and cross-references.

**Acceptance Criteria:**
- Organized by type: class, sequence, state, C4
- File path for each diagram
- Cross-reference to the ICD section where each diagram is embedded

### L2-3.15: Seam Selection for Generation

**Traces to:** L1-3

SeamQ shall support generating ICDs for a specific seam or all detected seams.

**Acceptance Criteria:**
- `seamq generate <seam-id>` generates for one seam
- `seamq generate --all` generates for all seams
- Each seam gets its own output directory

## L1-4: Diagram Generation

### L2-4.1: Class Diagram Generation

**Traces to:** L1-4

SeamQ shall generate PlantUML class diagrams for all 12 diagram categories defined in the PRD.

**Acceptance Criteria:**
- Generates API surface, backend contracts, backend controllers, datastore schema, domain data objects, frontend services, message interfaces, realtime communication, telemetry models, telemetry service, registration system, and file/storage class diagrams
- Each diagram is valid PlantUML syntax
- Diagrams show correct relationships (implements, extends, uses, composes)
- Diagrams fit on a letter-size page

### L2-4.2: Sequence Diagram Generation

**Traces to:** L1-4

SeamQ shall generate PlantUML sequence diagrams for all 15 sequence types defined in the PRD.

**Acceptance Criteria:**
- Generates app startup, plugin lifecycle, data consumption, tile add/subscribe, tile remove/unsubscribe, request flow, query flow, command flow, command-response UI, config CRUD, advisory message, telemetry subscribe, error handling, message bus routing, and review telemetry sequences
- Each diagram is valid PlantUML syntax
- Participants correctly represent the actual services/components detected
- Activation bars used appropriately

### L2-4.3: State Diagram Generation

**Traces to:** L1-4

SeamQ shall generate PlantUML state machine diagrams for detected stateful entities.

**Acceptance Criteria:**
- Generates datastore state machine diagrams
- Generates subscription lifecycle state machine diagrams
- States and transitions derived from detected state machine patterns
- Valid PlantUML state diagram syntax

### L2-4.4: C4 Diagram Generation

**Traces to:** L1-4

SeamQ shall generate C4 architecture diagrams using the C4-PlantUML standard library for all 12 C4 diagram types defined in the PRD.

**Acceptance Criteria:**
- Generates System Context, Context within Architecture, Container, Component (frontend services), Component (backend), Plugin API Layers, Plugin Architecture, Data Flow, Subscription Channel Map, Protocol Stack, Dynamic, and Deployment diagrams
- Uses `C4-PlantUML` macros (`System_Boundary`, `Container`, `Component`, etc.)
- Valid PlantUML syntax with C4 includes
- Technology labels included on containers and components

### L2-4.5: PlantUML Source Output

**Traces to:** L1-4

SeamQ shall output `.puml` source files for all generated diagrams.

**Acceptance Criteria:**
- Every diagram saved as a `.puml` file in the seam's `diagrams/puml/` or `diagrams/c4/` directory
- Files named by convention matching the PRD output structure
- Files contain complete, self-contained PlantUML source

### L2-4.6: Diagram Rendering

**Traces to:** L1-4

SeamQ shall render diagrams to SVG and PNG using a local or remote PlantUML server.

**Acceptance Criteria:**
- Supports local PlantUML JAR invocation
- Supports Docker-based PlantUML server
- Supports remote PlantUML server URL
- Renders to configured format (SVG, PNG, or both)
- Rendered files placed alongside `.puml` sources

### L2-4.7: Diagram Type Filtering

**Traces to:** L1-4

The `seamq diagram` command shall filter generated diagrams by the `--type` option, only producing diagrams that belong to the requested category.

**Acceptance Criteria:**
- `--type class` generates only class diagrams
- `--type sequence` generates only sequence diagrams
- `--type state` generates only state diagrams
- `--type c4-context` generates only C4 System Context diagrams
- `--type c4-container` generates only C4 Container diagrams
- `--type c4-component` generates only C4 Component diagrams
- `--type c4-code` generates only C4 code-level diagrams (PluginApiLayers, Dynamic)
- `--type context` treated as alias for `c4-context`
- When `--type` is omitted, all applicable diagrams are generated
- The `type` parameter is passed from the CLI handler to the renderer

### L2-4.9: Standalone Diagram Output

**Traces to:** L1-4

All generated PlantUML diagrams shall be self-contained standalone documents that can be rendered independently without network access or external includes. No `!include` directives referencing remote URLs shall appear in generated `.puml` files.

**Acceptance Criteria:**
- No generated `.puml` file contains `!include` directives
- C4 diagrams define needed macros inline (e.g., `System`, `Container`, `Component`, `Rel`, `System_Boundary`, `System_Ext`, `ContainerDb`, `Rel_D`, `RelIndex`) using PlantUML `!define` or `!procedure` syntax
- All generated diagrams render correctly with a vanilla PlantUML installation (no plugins, no network)
- Existing unit tests continue to pass after the change

### L2-4.8: ContractSurface Convenience Properties

**Traces to:** L1-4

The `ContractSurface` model shall expose convenience query properties for all `ContractElementKind` values used by diagram generators.

**Acceptance Criteria:**
- `Observables` returns elements with Kind == Observable
- `Properties` returns elements with Kind == Property
- `SignalInputs` returns elements with Kind == SignalInput
- `Routes` returns elements with Kind == Route
- `Actions` returns elements with Kind == Action
- `Selectors` returns elements with Kind == Selector
- Existing convenience properties (Interfaces, InjectionTokens, AbstractClasses, InputBindings, OutputBindings, Methods, Types) remain unchanged

## L1-5: CLI Interface

### L2-5.1: Scan Command

**Traces to:** L1-5

SeamQ shall provide a `seamq scan` command that scans workspaces and builds the seam registry.

**Acceptance Criteria:**
- `seamq scan <path1> [path2] [pathN]` scans provided paths
- `seamq scan` (no args) uses paths from config file
- Outputs summary: workspace name, project count, export count per workspace
- Outputs total seam count
- `--save-baseline <path>` saves scan result as baseline JSON

### L2-5.2: List Command

**Traces to:** L1-5

SeamQ shall provide a `seamq list` command showing all detected seams in a table.

**Acceptance Criteria:**
- Displays table with columns: ID, Seam Name, Type, Provider, Consumer(s), Confidence
- Supports `--type <type>` filter
- Supports `--provider <name>` filter
- Supports `--confidence <threshold>` filter
- Table formatting with proper alignment

### L2-5.3: Generate Command

**Traces to:** L1-5

SeamQ shall provide a `seamq generate` command for ICD generation.

**Acceptance Criteria:**
- `seamq generate <seam-id> [--format md|html|pdf|docx]` generates for one seam
- `seamq generate --all [--format md|html|pdf|docx]` generates for all seams
- Multiple `--format` flags produce multiple output formats
- Default format is `md`
- Outputs file paths of generated artifacts

### L2-5.4: Diagram Command

**Traces to:** L1-5

SeamQ shall provide a `seamq diagram` command for diagram generation.

**Acceptance Criteria:**
- `seamq diagram <seam-id>` generates all diagram types for a seam
- `seamq diagram <seam-id> --type <type>` generates specific diagram type
- `seamq diagram --all` generates all diagrams for all seams
- Type options: context, class, sequence, state, c4-context, c4-container, c4-component, c4-code
- Outputs file paths of generated diagrams

### L2-5.5: Inspect Command

**Traces to:** L1-5

SeamQ shall provide a `seamq inspect` command showing detailed contract surface for a single seam.

**Acceptance Criteria:**
- Displays seam metadata: Type, Provider, Consumers, Confidence
- Lists contract surface elements grouped by category (Interfaces, Injection Tokens, Abstract Classes, Types/Enums, Input/Output Bindings, Methods)
- Each element shows source file path and line number
- Formatted with indentation and grouping

### L2-5.6: Validate Command

**Traces to:** L1-5

SeamQ shall provide a `seamq validate` command checking consumer contract compliance.

**Acceptance Criteria:**
- `seamq validate <seam-id>` validates one seam
- `seamq validate --all` validates all seams
- Per-consumer output with checkmarks/crosses for each contract element
- Summary with error and warning counts
- Exit code 1 when validation errors exist

### L2-5.7: Diff Command

**Traces to:** L1-5

SeamQ shall provide a `seamq diff` command comparing against a baseline.

**Acceptance Criteria:**
- `seamq diff <baseline-path>` compares current scan against saved baseline
- Reports ADDED, MODIFIED, REMOVED contract elements per seam
- Summary with total change counts
- Exit code 0 for no changes, 1 for changes detected

### L2-5.8: Init Command

**Traces to:** L1-5

SeamQ shall provide a `seamq init` command generating a config file with interactive prompts.

**Acceptance Criteria:**
- Prompts for workspace paths, aliases, and roles
- Prompts for output directory and formats
- Generates `seamq.config.json` in current directory
- Validates workspace paths exist before writing config

### L2-5.9: Export Command

**Traces to:** L1-5

SeamQ shall provide a `seamq export` command outputting raw seam data as JSON.

**Acceptance Criteria:**
- `seamq export <seam-id> --format json` exports one seam
- `seamq export --all --format json` exports all seams
- JSON output includes complete contract surface data
- Output to stdout by default, `--output-dir` for file

### L2-5.10: Global Options

**Traces to:** L1-5

SeamQ shall support global CLI flags on all commands.

**Acceptance Criteria:**
- `--verbose` enables detailed logging output
- `--quiet` suppresses all output except errors
- `--no-color` disables ANSI color codes in output
- `--output-dir <path>` overrides the configured output directory
- `--config <path>` specifies a custom config file location

### L2-5.11: Help Text and Error Messages

**Traces to:** L1-5

SeamQ shall provide helpful auto-generated help text and clear error messages for all commands.

**Acceptance Criteria:**
- `seamq --help` shows all available commands with descriptions
- `seamq <command> --help` shows command-specific options and examples
- Invalid commands show a "did you mean?" suggestion
- Missing required arguments produce clear error messages with usage example
- Version displayed via `seamq --version`

### L2-5.12: Serve Command

**Traces to:** L1-5

SeamQ shall provide a `seamq serve` command launching a local web server to browse ICDs.

## L1-13: API Documentation Generation

### L2-13.1: Doc Command CLI

**Traces to:** L1-13

SeamQ shall provide a `seamq doc <paths...>` command that scans workspace(s) and generates API reference documentation.

**Acceptance Criteria:**
- `seamq doc ./my-app` scans and generates docs for the workspace at the given path
- `seamq doc` with no args uses paths from config file
- `--output-dir` overrides default output location
- Output directory defaults to `./seamq-output` if not specified
- Exit code 0 on success, 1 on partial failure, 2 on fatal error

### L2-13.2: Per-Project Output Structure

**Traces to:** L1-13

Documentation output shall be organized into a folder named after each library or application being documented.

**Acceptance Criteria:**
- Each project creates a folder named after the project (e.g., `api/`, `shared/`, `shell/`)
- Inside each folder, a single `README.md` file contains the complete documentation
- A `diagrams/` subfolder holds any PlantUML `.puml` files generated per type
- Folder names are sanitized for filesystem safety (no spaces, special characters replaced with hyphens)

### L2-13.3: Public Export Discovery

**Traces to:** L1-13

The doc generator shall identify all publicly exported types from each project by reading barrel files (public-api.ts, public_api.ts, index.ts) and only documenting symbols that are part of the public API surface.

**Acceptance Criteria:**
- Only symbols exported through a barrel file (public-api.ts, public_api.ts, or index.ts) are documented
- Non-exported symbols (internal classes, helper functions, private types) are excluded
- Wildcard re-exports (`export * from`) are followed to discover the actual symbols
- Named re-exports (`export { Foo, Bar } from`) restrict documentation to only those names

### L2-13.4: Public Member Filtering

**Traces to:** L1-13

The doc generator shall exclude non-public members (private, protected) from documentation. Only members that are accessible when consuming the library externally shall appear.

**Acceptance Criteria:**
- Members prefixed with `private` keyword are excluded
- Members prefixed with `protected` keyword are excluded
- Members starting with `_` (TypeScript convention for private) are excluded
- Members starting with `#` (ECMAScript private fields) are excluded
- `@Input()`, `@Output()`, signal inputs (`input()`, `model()`) are always included (they are inherently public)
- Constructor parameters not decorated with `@Inject` or `inject()` are excluded

### L2-13.5: JSDoc Extraction and Display

**Traces to:** L1-13

The doc generator shall extract existing JSDoc/TSDoc comments from source code and display them in the documentation.

**Acceptance Criteria:**
- `/** ... */` block comments preceding a declaration are extracted and included
- `@param` tags are parsed and displayed in a parameters table
- `@returns` / `@return` tags are displayed in the return type description
- `@deprecated` tags produce a visible deprecation notice in the doc
- `@example` tags are rendered as fenced code blocks
- Multi-line descriptions are preserved with formatting

### L2-13.6: Auto-Generated Documentation

**Traces to:** L1-13

When a publicly exported symbol lacks JSDoc comments, the doc generator shall produce synthetic documentation by inferring purpose from the type signature, name, and Angular conventions.

**Acceptance Criteria:**
- Classes decorated with `@Component` get: "Angular component with selector `<selector>`"
- Classes decorated with `@Injectable` get: "Injectable service provided in `<providedIn>`"
- Classes decorated with `@Directive` get: "Angular directive with selector `<selector>`"
- Classes decorated with `@Pipe` get: "Angular pipe with name `<name>`"
- Interfaces get: "Interface defining the shape of `<name>`"
- Enums get: "Enumeration defining `<name>` values"
- Type aliases get: "Type alias for `<typeExpression>`"
- InjectionTokens get: "Injection token for `<typeParam>`"
- Methods without docs get: "Method that accepts `<paramTypes>` and returns `<returnType>`"
- Properties without docs get: "Property of type `<type>`"
- Input bindings get: "Input binding of type `<type>`"
- Output bindings get: "Output event emitter"

### L2-13.7: Type Signature Inference

**Traces to:** L1-13

The doc generator shall infer and display input types (parameter types) and output types (return types) for all documented methods and properties.

**Acceptance Criteria:**
- Method signatures include parameter names and types: `methodName(param1: Type1, param2: Type2): ReturnType`
- If TypeScript types are not annotated, `any` is used as a placeholder
- Generic type parameters are preserved (e.g., `Observable<T>`)
- Union types are displayed verbatim (e.g., `string | null`)
- For `@Input()` bindings, the bound type is shown
- For `@Output()` bindings, the `EventEmitter<T>` type parameter is extracted and shown

### L2-13.8: Inline PlantUML Class Diagrams

**Traces to:** L1-13

The doc generator shall produce a PlantUML class diagram for each documented type (class, interface, enum) showing its members, relationships, and hierarchy.

**Acceptance Criteria:**
- Each class produces a `.puml` file in the project's `diagrams/` subfolder
- The diagram shows: class name, public properties with types, public methods with signatures
- Interfaces show all members
- Enums show all values
- `extends` relationships rendered as inheritance arrows
- `implements` relationships rendered as realization arrows
- Input/output bindings shown with stereotypes (`<<input>>`, `<<output>>`)
- Injected dependencies shown as dependency arrows to the injected type
- The Markdown documentation embeds the diagram as `![ClassName](diagrams/ClassName.puml)` (or `.png` if rendered)
- Diagrams use the `PlantUmlEncoder` builder from SeamQ.Renderer

### L2-13.9: Single Markdown Output File

**Traces to:** L1-13

Each project's documentation shall be a single self-contained Markdown file with a consistent structure.

**Acceptance Criteria:**
- Title: `# <ProjectName> API Reference`
- Table of contents with anchor links to each section
- Overview section with project type, export count, and summary table by category
- One H2 section per category (Classes, Interfaces, Enumerations, Type Aliases, Injection Tokens, Components, Services, Directives, Pipes)
- One H3 subsection per documented symbol containing: description, class diagram reference, members table (name, type, description), methods table (signature, return type, description), source location
- Footer with generation timestamp

### L2-13.10: DI Integration

**Traces to:** L1-13

The doc generator shall be registered in the DI container and wired into the CLI following the established patterns.

**Acceptance Criteria:**
- `IDocGenerator` interface added to `SeamQ.Core/Abstractions/`
- `DocGenerator` implementation added to `SeamQ.Generator/`
- `DocCommand` added to `SeamQ.Cli/Commands/` following the static factory pattern
- Registered in `GeneratorServiceCollectionExtensions` and `CommandBuilder`
- Uses `IWorkspaceScanner` for scanning (does not duplicate scan logic)
- Uses `PlantUmlEncoder` for diagram generation (does not duplicate diagram logic)
